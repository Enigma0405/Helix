# Organization Data Flow Structure

This document outlines the intended data flow for Enterprise Organization Data within Helix.

## Core Philosophy

Helix maintains a strict boundary between **Application Code** and **Customer Data**.
1. **Application Code** lives in `backend/src`.
2. **Customer Data** lives in `backend/organization_data`.

Data in `organization_data` is intended to be manually populated by the user (or deployment scripts) using actual enterprise datasets. **No fake or placeholder data is generated by the system.**

## The Data Flow Lifecycle

### Phase 1: Population
Administrators or automated deployment pipelines place enterprise JSON, Markdown, PDF, and other files into the respective subdirectories of `organization_data/<tenant>/organization_seed/` and `organization_data/<tenant>/historical_memory/`.

### Phase 2: Ingestion
The Helix Ingestion Pipeline (`backend/src/ingestion/`) scans the `organization_seed` and `historical_memory` directories.
- It parses, classifies, validates, and normalizes this raw data.
- It transforms the verified data into **Canonical JSON**.

### Phase 3: Manifest Generation
During and after ingestion, the pipeline outputs structured reports to the `manifests/` directory.
- Validation logs detailing what was successfully parsed.
- Checksums (SHA-256) ensuring idempotency, guaranteeing that duplicate files are not re-ingested.

### Phase 4: Runtime Execution
During live system usage, new evidence (e.g., from operators or systems like MES/LIMS) enters through the API and is temporarily tracked. For testing and simulations, such events may be loaded from `demo_evidence/`. This evidence triggers the Runtime Engine (`backend/src/runtime/`) to begin an Investigation Context, actively pulling from the already ingested Organization Memory.
